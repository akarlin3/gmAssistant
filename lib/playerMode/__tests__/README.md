# Player Mode tests

These tests guard the Player Mode **security boundary**: the projection that
redacts campaign content into the public-read share docs players read. A bug
here is a privacy leak, not a cosmetic defect.

## The redaction invariant (property-based)

`redaction.property.test.ts` proves ‚Äî across arbitrary adversarial campaign
state generated by `arbitraries.ts` ‚Äî that the real `buildSlotProjection` /
`buildPlayerGraph` never leak anything the GM hid. The properties:

| # | Invariant |
| --- | --- |
| I1 | No entity the GM hasn't made visible to a slot appears in that slot's projection (fail-closed: unconfigured ‚áí hidden). |
| I2 | No field that resolves `private` (default or per-instance override) survives on a visible entity. |
| I3 | Structural/non-content fields (`isPublic`, sidekick/*, ‚Ä¶) are never surfaced. |
| I4 | An edge is published only when **both** endpoints are themselves projected to the slot ‚Äî so it can never betray a hidden entity ‚Äî and only when its own visibility admits the slot and it isn't a review-queue edge. |
| I5 | GM-only edge fields (notes, visibility, customVisibleTo) are stripped. |
| I6 | A session-log entry appears only when its visibility admits the slot, with the internal `visibility` field stripped. |
| I7 | The PC-ownership bypass (full sheet to the owning slot) never leaks **another** slot's PC. |
| I9 | `buildPlayerGraph` is a pure consumer ‚Äî it adds no node/edge beyond the redacted projection. |

The map counterpart lives in `lib/maps/__tests__/playerProjection.property.test.ts`
(M1‚ÄìM5: only `visibleToPlayers` layers, on-layer features only, GM fields
stripped, maps with no visible layer omitted).

## Reuse

`arbitraries.ts` is the shared generator for both the redaction proof and the
write-back merge proof (`lib/crdt/__tests__/writeback-merge.property.test.ts`),
so the authored merge is tested against the same adversarial space as the
boundary it must not violate.

## What to do if a property fails

`fast-check` will print a shrunk counterexample ‚Äî a minimal campaign + slot that
leaks. That is a **real bug in the projection**, not the test. Fix
`lib/playerMode/projection.ts` (or `resolveVisibility.ts` / `playerProjection.ts`),
then pin the shrunk case as an explicit regression test alongside the unit tests
in `projection.test.ts`.

The unit tests (`projection.test.ts`, `resolveVisibility.test.ts`,
`graphProjection.test.ts`, `rules.test.ts`, ‚Ä¶) cover concrete scenarios and the
emulator-backed Firestore rules; the property tests cover the space between
them.
